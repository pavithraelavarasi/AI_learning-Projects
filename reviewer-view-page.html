Question 1

CSS Box Model Literacy

HTML + CSS

The Original Question

Objective: To develop CSS Box Model literacy by reverse-engineering an AI-generated personal profile card and making intentional modifications to observe how each property shapes layout and spacing.

Input

You have just met someone at a networking event, and you want a quick way for them to remember who you are later. You need a simple visual card that shows your face, name, what you do, and a short note about yourself. Build the simplest card that lets you see this at a glance.

Question Content

Expectations

  • Read the AI-generated HTML and CSS code line by line and understand what each part does.
  • Identify where margin, border, padding, and content appear in the code.
  • Explain how each property affects the card’s appearance and spacing.
  • Modify at least one property from each box model layer and observe the changes.
  • Explain the difference between padding and margin without looking it up.
  • Critique one design or spacing decision and improve it.

Reviewer Questions

  1. What is the purpose of this profile card, and how does that purpose influence your layout decisions?
  2. Show where margin, padding, border, and content appear in your code and explain each one in your own words.
  3. What happened when you changed the margin values, and how was that different from changing padding?
  4. Which box model property had the biggest effect on spacing, and why?
  5. How did the border affect the visual structure of the card?
  6. What design or spacing choice in the AI-generated version did you disagree with?
  7. What exact improvement did you make, and why is your version better?
  8. If another student confused padding and margin, how would you explain the difference using your card as the example?
Question 2

Responsive Layouts with Flexbox

Responsive CSS

The Original Question

Objective: To develop an understanding of responsive layouts using Flexbox by reverse-engineering an AI-generated profile card directory and experimenting with how layout properties behave across different screen sizes.

Input

You’re building a simple directory where you need to quickly scan multiple people at once. Show 4 profile cards in a single row on larger screens, but make sure it adapts neatly on smaller devices so everything stays readable. Build the simplest layout that does this.

Question Content

Expectations

  • Understand the HTML and CSS line by line and how the layout is structured.
  • Identify Flexbox properties such as display: flex, justify-content, align-items, gap, and flex-wrap.
  • Explain how each property affects spacing, alignment, and responsiveness.
  • Modify at least one Flexbox-related property and observe layout changes at smaller widths.
  • Experiment with the number of cards per row and describe how responsiveness changes.
  • Explain the difference between fixed and responsive layouts.

Reviewer Questions

  1. Why is Flexbox a good fit for this kind of profile directory layout?
  2. Which Flexbox properties are responsible for card alignment, wrapping, and spacing in your page?
  3. What happens when the screen width becomes smaller, and which CSS rules control that behavior?
  4. How did changing the number of cards per row affect readability and layout balance?
  5. What visual difference did you observe when adjusting gap or justify-content?
  6. How would you explain the difference between a fixed layout and a responsive layout using this example?
  7. What would break or feel awkward if the cards stayed in one rigid row on mobile?
  8. What change did you make that most improved the layout on smaller screens?
Question 3

Landing Page Structure and Sections

Page Layout

The Original Question

Objective: To develop an understanding of webpage structure and section-based layouts by reverse-engineering an AI-generated product landing page and experimenting with how different HTML and CSS properties affect presentation and usability.

Input

You’ve just created a new product and need a simple webpage to introduce it to people. Visitors should immediately understand what it is, see its key benefits, and have an easy way to reach out if they’re interested. Build the simplest page that includes a clear header, a section highlighting the main features, and a contact form at the bottom to collect their details.

Question Content

Expectations

  • Understand the HTML and CSS line by line and how the webpage is structured.
  • Identify where spacing, typography, color, section alignment, form styling, and button styling are used.
  • Explain how the header, features section, and contact form contribute to the user experience.
  • Modify layout, color, spacing, or typography and observe the visual effect.
  • Explain the difference between content structure and visual styling.

Reviewer Questions

  1. Why is it important for this page to have a clear header, feature section, and contact form in that order?
  2. Which part of your HTML defines structure, and which part of your CSS mainly controls appearance?
  3. How does the header help a visitor understand the product quickly?
  4. What makes the features section easy or difficult to scan?
  5. How does the contact form support the page’s overall goal?
  6. What change did you make to spacing, typography, or color, and what improvement did it create?
  7. How would you explain the difference between content structure and visual styling using your page?
  8. If you removed one section, which part of the user experience would suffer most, and why?
Question 4

Passenger Counter and Dynamic UI

JavaScript Interactivity

The Original Question

Objective: To develop an understanding of basic state management, user interaction handling, and dynamic UI updates in JavaScript by reverse-engineering an AI-generated passenger counter tool and experimenting with how data changes are reflected on the screen in real time.

Input

You are a conductor. Right now, passengers are boarding and alighting. You need to know if the bus is full. Build the simplest tool that helps you in that moment.

Question Content

Expectations

  • Understand the HTML, CSS, and JavaScript line by line and explain how the tool works internally.
  • Identify variables, functions, event listeners, and DOM manipulation in the code.
  • Explain how the passenger count updates when people board or leave.
  • Modify the logic, such as adding a reset button, changing the bus capacity, or preventing negative counts.
  • Experiment with features like “Bus Full” or remaining seats and observe the UI response.
  • Explain the difference between static webpages and interactive webpages.

Reviewer Questions

  1. What variable or state value represents the current passenger count in your program?
  2. Walk through what happens in the code when the user clicks the board or leave button.
  3. Where is the DOM updated, and how does the screen stay in sync with the data?
  4. What problem were you solving by adding or changing logic such as reset, capacity limits, or zero protection?
  5. How did you prevent invalid states, such as the count dropping below zero?
  6. What is the difference between a static page and this interactive tool?
  7. How would the user experience change if the passenger count updated only in a variable but not on the screen?
  8. Which feature most clearly demonstrated real-time UI updates to you?
Question 5

Digital Restaurant Board

Dynamic Content

The Original Question

Objective: To develop an understanding of basic user input handling, dynamic content rendering, and simple state updates in JavaScript by reverse-engineering an AI-generated digital restaurant board and experimenting with how information is displayed and updated on the screen.

Input

You own a small restaurant. Every morning you write the day's items on a board. Build the simplest digital version of that board.

Question Content

Expectations

  • Understand the HTML, CSS, and JavaScript line by line and explain how the board is structured and updated.
  • Identify input fields, buttons, arrays, event listeners, and DOM manipulation.
  • Explain how menu items are added, displayed, updated, or removed.
  • Modify functionality such as prices, styling, special-item highlighting, or clear-all behavior.
  • Experiment with the layout or number of displayed items and observe behavior on smaller screens.
  • Explain the difference between static HTML content and dynamically generated content.

Reviewer Questions

  1. How does your app take the user’s input and turn it into visible menu items on the board?
  2. Why is an array or list structure useful for this kind of problem?
  3. What role do event listeners play in adding or updating menu items?
  4. What change did you make to the board’s functionality, and what did that add for the user?
  5. How is dynamic content different from writing all menu items directly in HTML at the start?
  6. What happens to the interface when many items are added or when the screen becomes smaller?
  7. How would you explain the relationship between the stored data and what is rendered on the page?
  8. Which part of your implementation feels most reusable if you wanted to build a similar app later?
Question 6

Persistence with Browser Storage

localStorage

The Original Question

Objective: To develop an understanding of browser storage and data persistence by reverse-engineering an AI-generated restaurant menu management tool and experimenting with how saved data can remain available even after refreshing the page.

Input

You are managing your restaurant menu tool. You accidentally refresh the page and everything you entered is gone. How do you make sure that never happens again?

Question Content

Expectations

  • Understand the HTML, CSS, and JavaScript line by line and how the menu tool works.
  • Identify where localStorage, saving, retrieving, and updating stored values are used.
  • Explain how the application remembers data after refresh.
  • Modify storage logic or interface behavior and observe what happens after reloading.
  • Explain the difference between temporary in-memory data and persistent browser storage.

Reviewer Questions

  1. What data is saved in browser storage, and when is it written there?
  2. What steps happen when the page first loads and needs to restore saved information?
  3. How is localStorage different from keeping values only in JavaScript variables?
  4. What did you change in the storage logic or interface behavior, and why?
  5. How did you verify that your data really persists after a refresh?
  6. What would the user lose if storage were removed from this application?
  7. What kinds of data should or should not be stored this way in a real application?
  8. How would you explain persistence to someone who has never used browser storage before?
Question 7

Java OOP Basics

Classes and Objects

The Original Question

Objective: To develop object-oriented thinking by reverse-engineering an AI-generated Java solution and understanding how classes and objects give structure to data that was previously just an array.

Input

You’re browsing an online store, saving items you may want later, each with details like price, category, or notes. As your choices change, you update or remove them. Build the simplest tool to capture and manage these detailed entries.

Question Content

Expectations

  • Read the AI-generated Java code line by line and understand what each part does.
  • Identify where the class, fields, constructor, and methods appear.
  • Explain what an object is compared to a plain variable.
  • Modify at least one field, such as adding a category, and observe the effect on the class.
  • Explain the difference between a class and an object without looking it up.

Reviewer Questions

  1. What problem does using a class solve compared with storing everything in separate variables or arrays?
  2. Show the class, its fields, the constructor, and at least one method, and explain the role of each.
  3. How would you describe an object in your own words using your shopping item example?
  4. What field did you add or change, and what other parts of the class had to be updated because of it?
  5. How does the constructor help keep object creation consistent?
  6. What is the difference between the class itself and one object created from it?
  7. How would the code become harder to manage if you did not use objects here?
  8. What part of this exercise made object-oriented design feel more concrete to you?
Question 8

Inheritance and Code Reuse

Inheritance

The Original Question

Objective: To develop code reuse thinking by reverse-engineering an AI-generated class hierarchy and understanding how inheritance eliminates duplication when two things share common behavior.

Input

You already have a system to keep track of things you might want someday. Now you also need a list for things you’ve decided to buy. This new list behaves almost the same, but has one extra capability. Design the solution so both lists share what’s common while still allowing their differences.

Question Content

Expectations

  • Identify which code lives in the parent and which lives in the child.
  • Explain why the child class can use parent methods without rewriting them.
  • Modify one method in the parent and observe the effect on the child.
  • Add one new method to the child and confirm the parent remains unchanged.
  • Explain in one sentence what problem inheritance solved here.

Reviewer Questions

  1. Which behavior is shared between the parent and child classes, and why does that make inheritance useful here?
  2. Show which methods or fields belong in the parent and which belong only in the child.
  3. Why can the child class use code from the parent without duplicating it?
  4. What happened when you changed a parent method, and how did that affect the child?
  5. What new child-specific capability did you add, and why should it not live in the parent?
  6. How would the project become harder to maintain if you copied the same code into two separate classes?
  7. What limitation or caution should you keep in mind when using inheritance?
  8. In one clear sentence, what exact problem did inheritance solve in this exercise?
Question 9

Polymorphism and Interfaces

Abstraction

The Original Question

Objective: To develop abstraction thinking by reverse-engineering an AI-generated interface implementation and understanding how one method call can behave differently depending on which object receives it.

Input

You now have two kinds of lists, one for things you’re considering and one for things you’re ready to buy. Even though their purpose differs, the way you interact with them feels the same: you add items, remove them, and look through what’s there. Build your solution so you can use either list in exactly the same way.

Question Content

Expectations

  • Identify where the interface is declared and where it is implemented.
  • Explain why both classes must have the same method names.
  • Call the same method on both objects and observe different outputs.
  • Add a third class that implements the same interface without touching existing code.
  • Explain the difference between interface and inheritance in one sentence.

Reviewer Questions

  1. What contract does the interface define for the classes that implement it?
  2. Why do both classes need the same method names and signatures?
  3. Show one example where the same method call leads to different behavior depending on the object.
  4. What made it possible to add a third class without changing the existing calling code?
  5. How does polymorphism help the rest of the program stay simpler?
  6. What is the difference between saying “is a type of” and “can be used through this interface” in this exercise?
  7. How would you explain the difference between interface and inheritance to a beginner?
  8. What real advantage did abstraction provide in your design here?
Question 10

Final OOP Integration Project

Capstone Review

The Original Question

Objective: To consolidate all OOP concepts by reverse-engineering a complete AI-generated console application and being able to trace every design decision back to a concept learned during the week.

Input

You’re trying to keep your life organized, things you want someday, things you need to buy now, and tasks you must get done. Instead of juggling multiple tools, you want one simple place to manage everything. Build a console-based tool that helps you track, update, and review all these lists together, using the ideas you’ve been learning to bring it all into one cohesive system.

Question Content

Expectations

  • Identify which class uses encapsulation, which uses inheritance, and which uses an interface.
  • Trace one complete user action from input to object to display.
  • Modify one part of the app without breaking the rest.
  • Explain why all three lists can be handled through the same interface reference.
  • Explain what OOP gave this project that the JavaScript version from Day 5 did not.

Reviewer Questions

  1. Show where encapsulation, inheritance, and interface-based design appear in your final project.
  2. Trace one user action completely, from the moment the user enters data to the moment the output is shown.
  3. What part of the design made it possible to manage different lists inside one system?
  4. What change did you make without breaking the rest of the application, and why did the structure support that safely?
  5. Why can all three lists be handled through the same interface reference?
  6. What did OOP give this project that your earlier JavaScript version did not provide as clearly?
  7. If you had to extend this app next week, which part of the design would help you most?
  8. Which concept from the week feels most essential in the final project, and why?

This final section is especially useful for oral review because it checks both concept recognition and the student’s ability to connect design choices across the full project.
