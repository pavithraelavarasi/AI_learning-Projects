Why Semantic HTML?

Semantic tags tell the browser — and humans — what a piece of content is, not just how it looks. This makes your pages smarter, more accessible, and easier to maintain.

Div soup vs. semantic structure

A page built with only <div> tags looks the same visually, but has no meaning. Screen readers, search engines, and other developers have no idea what role each block plays. Semantic HTML replaces generic divs with meaningful tags.

Watch the transformation — click the buttons
◆ Semantic HTML is the vocabulary of the web. Each tag carries meaning that machines and humans can understand instantly — without reading the content inside.
Accessibility
Screen readers use tags to navigate. <nav>, <main>, <header> create automatic landmarks.
SEO
Search engines rank pages higher when they can understand the structure and hierarchy of content.
Cleaner Code
Semantic tags are self-documenting. <footer> is instantly clearer than <div class="footer">.
Team Friendly
Any developer can scan a semantic page and immediately understand where everything lives.

Lesson 03 of 08

<header> & <nav>

The header introduces your page or section. The nav holds your navigation links — and together they form the top of every great website.

What goes in <header>?

<header> typically holds the site logo, page title, or introductory content for a section. It's not just the top of the page — any <article> or <section> can also have its own <header>.

HTML
<header> <a href="/"> <img src="logo.svg" alt="MySite"> </a> <h1>MySite</h1> <p>A short tagline here</p> </header>
What goes in <nav>?

<nav> wraps primary navigation links only — the main menu. Not every group of links qualifies; footer links or breadcrumbs don't need to be inside a <nav>.

HTML
<nav> <ul> <li><a href="/">Home</a></li> <li><a href="/about">About</a></li> <li><a href="/blog">Blog</a></li> </ul> </nav>

<nav> purpose
Only for major navigation links. Use <ul> + <li> inside for accessibility.
Multiple navs
A page can have more than one <nav> — e.g. site-wide nav and a table of contents.
Landmark role
Screen readers expose <header> and <nav> as landmark regions for quick jumping.
Lesson 04 of 08

<article> & <section>

These two tags organise the content inside <main>. Article is for self-contained pieces. Section groups related content thematically.

The key difference

An <article> should make sense on its own — like a blog post, news story, or product card. A <section> groups related content that belongs together within a larger whole, like chapters in a book.

HTML
<article> ← self-contained post <header> <h2>My Blog Post Title</h2> <time datetime="2024-06-01">June 1, 2024</time> </header> <section> ← Introduction section <h3>Introduction</h3> <p>Opening paragraph…</p> </section> <section> ← Next section <h3>The Main Point</h3> <p>More content…</p> </section> </article>

✓ Simple test for <article>: Could you publish this block independently on another site? If yes → <article>. If it needs the surrounding page to make sense → <section> or plain <div>.
<article> examples
Blog posts, news items, forum posts, product cards, comments — anything independently shareable.
<section> examples
Chapters in a document, tabs in a UI, thematic groupings within a larger page.
Nesting
<article> can contain <section>s. A <section> can contain <article>s (e.g. a comments section).
Headings required
Every <section> and <article> should have a heading so screen readers can navigate between them.

More Semantic Tags

Beyond the big five structural tags live a set of content-level semantic elements that give specific meaning to the pieces inside your articles and sections.

<figure> + <figcaption>

Machine-readable dates and times via the datetime attribute.
<address>

Lesson 07 of 08

Web Accessibility

Accessible HTML ensures that your pages work for everyone — including people using screen readers, keyboard navigation, or low-vision displays. These 10 rules form your accessibility foundation.

RULE 01
Use Semantic HTML Elements
Use the right tag for the right purpose. Semantic elements carry built-in accessibility roles — no extra ARIA needed.
Avoid
<div onclick="...">Submit</div>
No role, no keyboard access.
Better
<button>Submit</button>
Focusable, activatable by keyboard.
Use <header>, <nav>, <main>, <section>, <article>, <footer> instead of generic <div> everywhere. Never skip heading levels — <h1> → <h2> → <h3> in order.
RULE 02
Add Alt Text to Every Image
Every <img> must have an alt attribute.
Descriptive
alt="A student reading a book"
Decorative
alt="" — empty, not missing
Never
alt="image1.jpg" — filename is not a description
RULE 03
Proper Heading Hierarchy
One <h1> per page (the main title). Subheadings follow in order: <h2>, <h3>, etc. Never skip levels — going from <h1> directly to <h4> breaks the document outline for screen readers.
h1 — Page Title (one per page)
h2 — Main Section
h3 — Subsection
h4 — Sub-subsection
h2 — Next Main Section
RULE 04
Label All Form Inputs
Every input must have an associated <label>. The for attribute on the label must match the id on the input.
HTML
<label for="email">Email address</label> <input type="email" id="email" name="email">
RULE 05
Sufficient Color Contrast
Text must be readable against its background. Minimum contrast ratio: 4.5:1 for normal text (3:1 for large text). Never rely on color alone to convey meaning.
3.0:1 Fail This grey text on white fails the contrast check. Hard to read for low-vision users.
15.3:1 Pass Dark text on white. Clear for almost all users including those with low vision.
RULE 06
Keyboard Navigation
All interactive elements must be reachable with the Tab key. Never remove the default focus outline without a replacement.
Links are also keyboard-accessible — this one has a focus ring tab here
Rules 7–10 at a glance
RULE 07
Meaningful Link Text
Never <a>Click here</a>. Links must describe their destination: <a>Read about HTML semantics</a>. Screen readers list all links out of context.
RULE 08
Language Attribute
Always declare <html lang="en">. Screen readers use this to choose the correct voice and pronunciation engine. Without it, content may be mispronounced.
RULE 09
Use <title> in <head>
Every page needs <title>My Portfolio</title>. It is the first thing a screen reader announces when the page loads. Tab titles also help orientation.
RULE 10
Pair Visual Cues with Text
Never rely on color or icons alone. An error turning red must also say "Error: This field is required." Color blindness affects 1 in 12 men.
